# Звіт про порівняння Docker-образів для ML-сервісу

## Показники розміру та продуктивності

| Характеристика | Fat Image (`ml-service:fat`) | Slim Image (`ml-service:slim`) |
| :--- | :--- | :--- |
| **Загальний розмір** | **8.23 GB** | **1.03 GB** |
| **Базовий образ** | `python:3.9` (повний Debian) | `python:3.9-slim` (мінімальний) |
| **Стратегія збірки** | Одностадійна (Single-stage) | Багатостадійна (Multi-stage) |
| **Дистрибутив PyTorch** | Стандартний (з CUDA/GPU) | Тільки CPU (Оптимізований) |

## Детальний аналіз

### 1. Чому "Fat" образ такий величезний (8.23 ГБ)?
* **Бібліотеки CUDA:** Команда `pip install torch` за замовчуванням завантажує важкі
 бінарні файли для підтримки NVIDIA GPU, що додає приблизно 4-5 ГБ даних, які не використовуються в середовищі без відеокарти.
* **Інструменти збірки:** Включає повні системні заголовки, компілятори (`gcc`), 
 документацію та кеш менеджера пакетів `apt`.
* **Зайві шари:** Кожна операція в Dockerfile створювала постійний шар,
 який не було очищено від тимчасових файлів.

### 2. Як "Slim" образ досяг скорочення на ~87.5%?
* **Multi-stage build:** Ми використали етап `builder` для підготовки середовища,
 а у фінальний образ скопіювали лише готову віртуальну папку `.venv` без компіляторів.
* **CPU-only Wheels:** Використання `--extra-index-url https://download.pytorch.org/whl/cpu`
 дозволило уникнути встановлення CUDA.
* **Мінімальна база:** Використання `-slim` версії Python видаляє системні утиліти,
 які не потрібні для виконання коду.

## Пропозиції щодо подальшої оптимізації
1. **Квантизація моделі:** Конвертація моделі в формат INT8 зменшить файл `.pt` з 14.5 МБ приблизно до 5 МБ.
2. **Перехід на ONNX:** Використання `onnxruntime` замість `torch` дозволить
 зменшити розмір образу до **200-300 МБ**.
3. **Distroless образи:** Використання Google Distroless повністю видалить
 оболонку (shell) та менеджер пакетів, що зробить контейнер ще легшим та безпечнішим.
